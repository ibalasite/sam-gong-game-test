3. 技術架構概覽

  • Client: Cocos Creator 3.8.x（Web + 手機跨平台），TypeScript 開發
  • Server: Colyseus 0.15.x（Node.js/TypeScript），@colyseus/schema 做狀態同步
  • 架構原則: Server-authoritative — Schema 類別僅含欄位定義，遊戲邏輯獨立於 State
  • 持久層: PostgreSQL 16.x（玩家資料、牌局紀錄）+ Redis 7.x（matchmaking queue、session cache）
  • 部署: Kubernetes (k8s)；Client 走 WebSocket + TLS

4. 三公遊戲規則（D-Series）

編號規則
D1每局發牌：莊家 + 閒家各發 3 張牌（標準 52 張牌組）
D2點數計算：10、J、Q、K 算 0 點；A 算 1 點；2-9 依面值
D3手牌點數：3 張牌總點數 mod 10 為最終點數
D4三公（Sam Gong）：3 張牌均為 10/J/Q/K，點數為 0 但為最大手牌
D5比牌：高點數勝；同點比花色大小（黑桃 > 紅心 > 方塊 > 梅花）
D6結算方向：閒家贏莊家 → 莊家付賠；莊家贏閒家 → 閒家付注
D7Rake：floor(pot × 0.05)，最少 1 籌碼，從底池抽取
D8平手（同點同花色）：A 為最小牌，依 2-9-10(0)-J-K-Q 花色排列
D13莊家破產：籌碼不足以支付所有贏家時，先到先得算法分配

7. 已知風險

  • R-Legal（台灣法規）：三公屬「博弈」性質遊戲。本產品必須採「虛擬籌碼、不可兌換、不可購買」模式以規避賭博罪嫌疑。
  • R-Cheat：卡牌遊戲最大技術風險為作弊（偷看牌、結果預測、外掛注入）。Server-authoritative 是基本功。
  • R-Scale：Colyseus 單實例約支援 1,000-3,000 同時連線；若目標規模 >10,000，需提早規劃 horizontal scaling。
  • R-UX：賭場風格 + 像素美術在跨平台（手機小螢幕）易顯擁擠；UI 需響應式設計。
