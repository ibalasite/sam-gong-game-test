2. Game State Machine

Game State Machine — 6 phases with transition conditions
stateDiagram-v2
    [*] --> waiting : onCreate (phase=waiting)

    waiting --> dealing : players.size >= 2
    waiting --> [*] : 60s 無人加入 自動解散

    dealing --> banker_bet : 發牌完成 (每人3張)
    dealing --> waiting : 發牌中玩家數 < 2

    banker_bet --> player_bet : 莊家下注確認 或 30s 超時
    banker_bet --> dealing : 莊家籌碼 < min_bet 跳過

    player_bet --> showdown : 所有閒家行動完成
    player_bet --> showdown : 所有閒家均 Fold

    showdown --> settled : Server 比牌完成 廣播 showdown_reveal
    showdown --> settled : all_fold 直接結算

    settled --> dealing : 籌碼更新完成 5s後 players.size >= 2
    settled --> waiting : players.size < 2
    settled --> [*] : 所有玩家離開

4. Settlement Logic Flowchart

Settlement Flowchart — Rake, Conservation, Banker Insolvency
flowchart TD
    START([開始結算]) --> VALIDATE{Phase === showdown?}
    VALIDATE -->|否| ERR[拋出錯誤]
    VALIDATE -->|是| CHECK_FOLD{all_fold?}

    CHECK_FOLD -->|是| ALL_FOLD[pot=0, rake=0\n莊家 escrow 退回\n所有 net_chips=0]
    ALL_FOLD --> PERSIST

    CHECK_FOLD -->|否| COMPARE[HandEvaluator.compare\n每位閒家 vs 莊家]
    COMPARE --> CLASSIFY[分類: winners/losers/ties/folders]
    CLASSIFY --> CALC_POT[pot = sum of losers' called_bets]
    CALC_POT --> BANKER_CHECK{莊家籌碼足夠?}

    BANKER_CHECK -->|是| NORMAL[逐一支付贏家\n三公 N=3, 9點 N=2, 1-8點 N=1]
    BANKER_CHECK -->|否| INSOLVENT[D13 先到先得算法\n依 seat_index 順序支付]

    NORMAL --> RAKE
    INSOLVENT --> RAKE
    RAKE{pot > 0?} -->|是| CALC_RAKE[rake=max(floor(pot*0.05), 1)]
    RAKE -->|否| ZERO_RAKE[rake=0]

    CALC_RAKE --> CONSERVATION
    ZERO_RAKE --> CONSERVATION
    CONSERVATION{sum net_chips + rake === 0?} -->|違反| ALERT[回滾事務\nPagerDuty CRITICAL]
    CONSERVATION -->|通過| PERSIST[PostgreSQL SERIALIZABLE\nUPDATE chip_balance\nINSERT chip_transactions\nINSERT game_sessions]

    PERSIST --> BROADCAST[廣播 settlement_result]
    BROADCAST --> DONE([結算完成])

    style ALERT fill:#f85149,color:#fff
    style INSOLVENT fill:#d29922,color:#fff

5. Anti-Addiction Flowchart

Anti-Addiction — Adult (soft limit) vs Minor (hard stop)
flowchart TD
    JOIN([玩家加入房間]) --> CHECK_MINOR{is_minor?}

    CHECK_MINOR -->|否 成人| ADULT[讀取 Redis aa:session:player_id]
    CHECK_MINOR -->|是 未成年| MINOR[讀取 aa:daily:player_id]

    ADULT --> ADULT_CHECK{session_play_seconds >= 7200?}
    ADULT_CHECK -->|否| PLAY[正常遊戲\n累積計時]
    ADULT_CHECK -->|是| WARN[發送 anti_addiction_warning\n等待確認]
    WARN --> CONFIRM{確認?}
    CONFIRM -->|是| RESET[重置 session_play_seconds\n繼續遊戲]
    CONFIRM -->|60s 超時| WARN

    MINOR --> MINOR_CHECK{daily_play_seconds >= 7200?}
    MINOR_CHECK -->|否| PLAY
    MINOR_CHECK -->|是| MINOR_SIGNAL[發送 anti_addiction_signal]
    MINOR_SIGNAL --> ROUND{牌局進行中?}
    ROUND -->|是| WAIT[等待本局結算後]
    ROUND -->|否| KICK
    WAIT --> AFTER[本局 settled]
    AFTER --> KICK[client.leave(4004)\nWS Close Code 4004]

    PLAY --> PERSIST[每局結算後\n持久化計時至 PostgreSQL]

    style KICK fill:#f85149,color:#fff
    style WARN fill:#d29922,color:#fff

6. JWT Authentication & Refresh Flow

JWT Auth — RS256, Token Rotation, 60s Ban Blacklist
sequenceDiagram
    participant C as Client
    participant API as REST API / Auth
    participant DB as PostgreSQL
    participant REDIS as Redis 7.x

    Note over C,REDIS: Access Token 過期 - Refresh Flow
    C->>API: POST /auth/refresh { refresh_token }
    API->>DB: SELECT refresh_tokens WHERE token_hash=SHA256(rt) AND revoked=false

    alt Refresh Token 無效/已撤銷
        DB-->>API: 0 rows
        API->>DB: 撤銷所有該帳號的 refresh_tokens (Token Reuse Detection)
        API-->>C: 401 token_invalid
    else Refresh Token 有效
        DB-->>API: token record
        API->>DB: UPDATE SET revoked=true (舊 token 撤銷)
        API->>API: 簽發新 Access Token RS256 TTL=1h
        API->>DB: INSERT 新 refresh_token TTL=7d
        API-->>C: 200 { access_token, refresh_token }
    end

    Note over API,REDIS: 封號後 60s 黑名單機制
    API->>DB: UPDATE users SET is_banned=true
    API->>REDIS: SETEX block:player_id 60 1
    API->>REDIS: DEL session:player_id
    API->>REDIS: PUBLISH sam-gong:admin:force_disconnect player_id

10. Rescue Chips Flowchart

Rescue Chips — Eligibility check, daily limit, atomic DB operation
flowchart TD
    REQ([玩家請求救援籌碼]) --> CHECK_BAL{chip_balance < 500?}
    CHECK_BAL -->|否 籌碼足夠| DENY1[rescue_unavailable\n籌碼餘額充足]
    CHECK_BAL -->|是| CHECK_DAILY{今日已領取?\ndaily_rescue_claimed_at === today}
    CHECK_DAILY -->|是| DENY2[rescue_chip_claimed\n今日已使用]
    CHECK_DAILY -->|否| CHECK_LOCK{Redis rescue:player_id:date\n分散式鎖}
    CHECK_LOCK -->|鎖存在| DENY3[並發重複請求\n忽略]
    CHECK_LOCK -->|無鎖| ACQUIRE[SET rescue:player_id:date NX EX 86400\n取得分散式鎖]
    ACQUIRE --> DB_TXN[PostgreSQL 事務\nUPDATE users SET chip_balance=100000\nSET daily_rescue_claimed_at=today\nINSERT chip_transactions rescue_chip]
    DB_TXN --> SUCCESS[發送 rescue_chips 訊息\n{ chips_received:100000, new_balance:100000 }]
    SUCCESS --> DONE([完成])

    style DENY2 fill:#30363d,color:#8b949e

12. Game Round Full Sequence

Full Game Round — dealing → banker-bet → player-bet → showdown → settled
sequenceDiagram
    participant C1 as Client Banker
    participant C2 as Client Player
    participant COL as Colyseus (SamGongRoom)
    participant EVAL as HandEvaluator
    participant SETTLE as SettlementEngine
    participant DB as PostgreSQL (SERIALIZABLE)

    Note over C1,DB: Phase: dealing
    COL->>COL: buildDeck + shuffle (Fisher-Yates, crypto.randomInt)
    COL->>C1: send('myHand', [card1, card2, card3]) 私人訊息
    COL->>C2: send('myHand', [card1, card2, card3]) 私人訊息
    COL->>COL: phase → banker-bet, action_deadline = NOW+30s

    Note over C1,DB: Phase: banker-bet (30s)
    C1->>COL: onMessage('banker_bet', { amount: 1000 })
    COL->>COL: 驗證 phase, balance >= 1000
    COL->>COL: banker chip_balance -= 1000 (escrow)
    COL->>COL: phase → player-bet

    Note over C1,DB: Phase: player-bet (30s each)
    C2->>COL: onMessage('player_call', {})
    COL->>COL: C2 chip_balance -= 1000 (escrow)
    COL->>COL: all acted → phase → showdown

    Note over C1,DB: Phase: showdown
    COL->>EVAL: compare(bankerHand, playerHand)
    EVAL-->>COL: player wins (9pt > 5pt)
    COL->>SETTLE: settle(input)
    SETTLE-->>COL: { winner: C2 net=+1000, loser: C1 net=-1000, rake: 50 }
    COL-->>C1: broadcast('showdown_reveal', { all hands })
    COL-->>C2: broadcast('showdown_reveal', { all hands })

    Note over C1,DB: Phase: settled
    COL->>DB: BEGIN SERIALIZABLE
    Note over DB: UPDATE chip_balance C1, C2
    Note over DB: INSERT chip_transactions x3 (win/loss/rake)
    Note over DB: INSERT game_sessions snapshot
    Note over DB: COMMIT
    DB-->>COL: committed
    COL-->>C1: broadcast('settlement_result')
    COL-->>C2: broadcast('settlement_result')
    Note over COL: 5s → resetForNextRound → phase=dealing
